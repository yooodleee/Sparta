# Git / GitHub 브랜치 컨벤션

이 문서는 팀의 버전 관리 규칙, 브랜치 전략, 네이밍 컨벤션을 정의한다.

---

## 1. 협업 규칙

- **메인 브랜치 보호**: 개별 `issue` 브랜치에서 `main`으로의 직접 push는 엄격히 금지한다.
- **커밋 단위 세분화**: 커밋은 단일 기능 단위로 나누어 빈번하게 수행하여 작업 이력을 명확히 관리한다.
- **동기화 필수**: merge 직후 모든 작업자는 `git pull`로 로컬을 최신 상태로 갱신한 뒤 다음 작업을 진행한다.
- **PR 기반 병합**: 모든 작업은 PR 생성 후 팀원 승인을 거쳐 `dev` 브랜치에 merge한다.
- **상세한 작업 기록**: issue 해결 시 변경 사항과 해결 방법 등 작업 내용을 구체적으로 기록한다.
- **리소스 정리**: merge 완료된 issue는 즉시 close 처리하고, 브랜치는 삭제한다.

---

## 2. 브랜치 유형

| 브랜치     | 역할                                            |
|---------|-----------------------------------------------|
| `main`  | 최종 배포용. 제출 및 리더보드 업로드가 가능한 상태만 유지. 직접 작업 금지.  |
| `dev`   | 통합 개발. 개별 기능(issue 브랜치)이 병합되는 중간 단계.          |
| `issue` | 실제 개발 공간. 반드시 하나의 issue에 대응하여 생성. `dev`에서 분기. |

**병합 방향**: `main` ← `dev` ← 개별 `issue branch`

### 브랜치 생성/병합 유의사항

- 작업할 사항이 있다면 **먼저 issue로 내용을 등록**한 뒤 `dev` 브랜치에서 새 issue 브랜치를 생성한다.
- **issue 라벨**을 붙이면 각 작업 공간(GPU 서버, 로컬 등)에서 브랜치를 식별하기 좋다.
- 작업 완료 후 더 이상 필요하지 않은 issue 브랜치는 반드시 삭제하거나 `dev` 브랜치로 병합한다.

## 3. Task 유형

| 유형         | 설명                                                            |
|------------|---------------------------------------------------------------|
| `data`     | 전처리, feature engineering, 데이터 포맷 변경                           |
| `feature`  | 새 기능 추가 (모델 구현, 에이전트 구현, K-fold 등)                            |
| `bug`      | 버그 수정, 잘못된 로직, 오류/예외 처리 수정                                    |
| `refactor` | 기능 변경 없는 코드 구조 개선 (가독성 향상, 중복 제거 등)                           |
| `config`   | `requirements.txt`, `pyproject.toml`, `config.yaml` 등 환경설정 문서 |
| `chores`   | 코드에 영향을 주지 않는 변경 (오타, 주석, 변수명, README 등)                      |

## 4. 네이밍 규칙

모두 **영문 소문자**로 작성한다. `작업내용요약`은 **동사-목적어** 형태로 2~4단어 이내.

| 항목         | 형식                                  | 예시                                         |
|------------|-------------------------------------|--------------------------------------------|
| 브랜치 이름     | `[TASK 유형]/[ISSUE 번호]-[작업내용요약]`     | `feature/12-develop-bert4rec`              |
| Issue 제목   | `[TASK 유형]: [작업내용요약]`               | `feature: develop bert4rec`                |
| Commit 메시지 | `[TASK 유형]: [작업내용요약] (#[ISSUE 번호])` | `data: preprocess news crawling data (#7)` |
| PR 제목      | commit 메시지와 동일                      | `data: preprocess news crawling data (#7)` |

---

## 5. 저장소에 올리지 말아야 할 것

- 모델 hyperparameter 튜닝 결과, 실험 결과 표
- 실험 산출물 (log, submits 등)
- 민감한 파일 (`.env`, GitHub/API 토큰, WandB 키 등)
